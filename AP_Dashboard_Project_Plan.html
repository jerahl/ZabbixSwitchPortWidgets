Reuse S
Document XIQ API calls — radio & wireless metrics
Enumerate and validate endpoints against a live AP_305C. Confirm exact response field names, units, and pagination behavior before implementation begins.

GET /xapi/v1/monitor/devices/{id}/wifi-data — per-radio channel utilization, noise floor, retry rate, TX power, channel, channel width
GET /xapi/v1/monitor/devices/{id}/clients — associated clients with RSSI, data rate, band, SSID, connected duration
GET /xapi/v1/monitor/devices/{id}/interface-stats — uplink in/out, errors (confirm if available; Zabbix IF-MIB fallback if not)

Key questions to answer: Does XIQ expose historical time-series for wireless metrics, or only point-in-time snapshots? If point-in-time only, Live Telemetry sparklines for channel util / noise must use Zabbix calculated items or show current value only. Confirm rate limits and recommended polling cadence.
XIQ API M
Document XIQ API calls — device config & identity
GET /xapi/v1/devices/{id} — model, serial, IQ Engine firmware + build, config type, management IP, MAC, NTP server, last config push timestamp, network policy ID
GET /xapi/v1/network-policy/{id} — policy name, list of assigned SSID IDs

Map every field to the exact System Info / Network Info / SSID table rows in the dashboard design. Note any fields that are missing from XIQ and must come from Zabbix host inventory instead.

Validate which metrics XIQ does not expose and must stay as Zabbix SNMP items. Expected Zabbix-only items: ICMP latency + packet loss, LLDP neighbor, PoE draw (verify against XIQ), uplink interface link state. These are the only items the existing Zabbix template needs to cover — no new SNMP items should be added if XIQ covers it.

M1 — ExtremeCloud IQ Integration Library
Weeks 3–5 · critical path · blocks M2, M3, M4
3 wk ▶
M1 expanded from 2 to 3 weeks. Wireless metrics and client data moved here from the old SNMP template tasks — XIQ now owns all wireless data.
Build XIQ OAuth2 PHP client — shared library
Bearer token request against POST /oauth2/token. Refresh loop with expiry tracking. Token cache in APCu (preferred) or filesystem fallback. Handle 401 → refresh → retry transparently. Build as a shared module imported by all AP widgets — never duplicate auth logic per widget. Store client credentials in a Zabbix macro or widget-level config field, not hardcoded.

Extract: model, serial, IQ Engine firmware + build, config type, management IP, MAC, NTP server, last config push timestamp, network policy ID. Map to System Info and Network Info KV table rows.
XIQ API S
XIQ: network policy + SSID config
GET /xapi/v1/network-policy/{id} → policy name, SSID ID list
GET /xapi/v1/ssid/{id} per SSID → VLAN, auth method, encryption, enabled bands
Cache aggressively (TTL ~5 min) — policy config changes infrequently. Feeds SSID Broadcast table on Wireless tab and System Info network policy row.
XIQ API M
XIQ: per-radio wireless metrics
GET /xapi/v1/monitor/devices/{id}/wifi-data
Extract per radio: channel, channel width, TX power, channel utilization, noise floor, retry rate, associated client count. Map radio index to 2.4 GHz / 5 GHz band labels. Feeds Radio cards on Wireless tab and Live Telemetry overview. No SNMP template extension required.
XIQ API M
XIQ: associated client list
GET /xapi/v1/monitor/devices/{id}/clients
Extract per client: MAC, hostname, SSID, RSSI, data rate, band, connected duration, radio index. Joined on MAC address with PacketFence node records on the Clients tab to produce the enriched client row. Eliminates the need for an AP SNMP association table walk.

XIQ: location + floor plan metadata
GET /xapi/v1/location/{id} → building name, floor, coordinates
Attempt GET /xapi/v1/location/{id}/floor-plan — if XIQ returns a usable format, use it in the Device sidecar (M5). Otherwise fall back to a static SVG asset per building stored as a Zabbix host inventory field.
