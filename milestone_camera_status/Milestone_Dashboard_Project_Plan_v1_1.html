Six tabs: Overview · Channels · Storage · Network · Events · Configuration.
  • iDRAC OOB integration on Dell DVRs. The Server Detail mockup surfaces inlet temperature, chassis service tag, RAID rebuild state, hot-spare count, and a dedicated iDRAC management interface row in the network table. None of that is reachable from the in-OS Zabbix Agent — it lives on the iDRAC BMC, accessible via SNMP (Dell iDRAC-MIB-SMIv2) or Redfish (HTTPS REST API). M0 adds an iDRAC capability probe; M1 adds a Dell iDRAC by SNMP sub-template conditionally linked off the per-RS host_prototype where chassis vendor is Dell.
  • Source-of-truth tagging. Every cell in the mockups carries a SourceBadge: zbx (data from a Zabbix-polled item — agent / SNMP / ICMP / HTTP) or ext (data from the Milestone API via the Site host). Widget design enforces this at render time: the SourceBadge primitive accepts src="zbx"|"ext"|"idrac" and renders a constant micro-pill next to every metric, so an operator viewing a number always knows which probe path produced it. Don't bury this — it's the difference between "Milestone says the camera is recording" and "the camera responds to ICMP." Operators need both, and need to know which is which when one says yes and the other says no.
  • Information-surface inventory work — what info can we actually pull? v1.0 sketched the templates from the existing YAML; v1.1 makes "what does each device let us measure" an explicit M0 deliverable. Three new M0 tasks: a camera-side surface matrix (per vendor: ONVIF, vendor-specific SNMP MIBs, vendor HTTP/REST endpoints, RTSP probe), a server-side surface matrix (Zabbix Agent vs WMI vs iDRAC SNMP vs iDRAC Redfish vs Milestone API — with the overlap zones flagged so we don't double-monitor and don't accept conflicting values without picking a canonical), and an iDRAC pilot probe (snmpwalk + Redfish curl on one Dell DVR). Output feeds directly into the M1 template structure.
Project context — what exists today, what we're building: The repo already contains Milestone_by_HTTP_API.yaml — a single-host template that polls the XProtect API Gateway (sites, recording servers, cameras, license, ESS WebSocket state) and exposes per-camera and per-RS data as dependent items on the gateway host. It works, and ships with a serviceable native dashboard (honeycomb + item navigator + item card). This project is the next step: split that one-host model into a three-tier host topology so cameras and DVRs get first-class Zabbix hosts that can be polled directly, and replace the native dashboard with custom widgets that match the apdetail visual language from AP Plan v3.2.
  • The Milestone API Gateway is still the source of truth. It always will be — it's the only thing that knows the camera ↔ hardware ↔ recording-server graph and the ESS state. The "Milestone Site" Zabbix host keeps every existing polling item: milestone.sites.get, milestone.rs.getall, milestone_cameras_read.sh[3600], milestone_ess_read.sh[], milestone.license.get, plus the API Gateway TCP probe. Nothing breaks.
  • Cameras become individual Zabbix hosts via a new host_prototype off the Camera LLD. Each host gets ICMP polled directly from the proxy (already an item prototype today — moves to the per-host template). Cameras with vendor SNMP support (Axis / Hanwha / Bosch) gain a vendor-specific sub-template selected by sysObjectID discovery. The Site host's per-camera dependent items (alarm, channel, hwmodel, ESS state breakdowns) stay where they are — they're cheap, they share the same source data, and ripping them out adds risk for no payoff. The new per-camera hosts add direct telemetry; they don't replace the API-derived items.
  • Recording Servers become individual Zabbix hosts with the Zabbix Agent installed on Windows. The agent lights up the OS layer — CPU, memory, disk, NIC, Windows services, event log, perf counters — that the Milestone API simply doesn't expose. Service-state monitoring on Milestone XProtect Recording Server, Milestone XProtect Event Server, Milestone XProtect Log Server. Storage path discovery via the Milestone API stamps per-volume LLD macros onto the agent host so disk-free items track each archive root directly.
  • Three custom widgets in the same module pattern as apdetail: mssite (site-level overview — license, RS roll-up, camera honeycomb, ESS health, recent events), mscamera (per-camera detail driven by _hostid broadcast — identity, ICMP sparklines, ESS state breakdown, recording state, optional snapshot thumbnail), msserver (per-DVR detail — agent metrics rings, services strip, storage volumes, attached-camera roll-up). Same Zabbix 7.4 widget framework and the same TcsAp...-style namespace prefix that AP Plan v3.2 settled on after G16.

§A — Existing-asset inventory Reference only — not a milestone
What's already in Milestone_by_HTTP_API.yaml + the six external scripts. Read this before M1 begins so the rework decisions in M1 don't accidentally regress something that already works.
A1 — OAuth2 password grant via /IDP/connect/token. Existing JS Script items get a token inline per call (no macro storage). Token is short-lived; no refresh dance. Keeps secret out of macros. Keep this pattern in the Site host.
A2 — Python-prepped camera snapshot via cron. milestone_cameras_state.py runs every 15m (cron) → writes a JSON file shaped {"<cam-guid>": {address, displayName, enabled, channel, mac, hwId, hwName, hwModel, lastModified}, "__array": […]}. Zabbix's external item milestone_cameras_read.sh[3600] just cats it. Reshape was moved to Python because Duktape's 10-second preprocessing timeout broke at 2500 cameras. Don't move it back.
A3 — ESS WebSocket dump via cron. milestone_ess_state.py runs once a day, opens a WebSocket to the Event & State Service, calls get_state, writes per-camera state JSON. Zabbix reads via external script. Daily cadence is intentional — at scale this takes 1–2 minutes. Per-camera dependent items extract individual fields with 7d DISCARD_UNCHANGED_HEARTBEAT.
A4 — Camera LLD with five LLD macros. {#CAM.ID} (GUID), {#CAM.NAME} (displayName), {#CAM.ADDRESS} (parent hardware IP, injected by the Python helper), {#CAM.MAC}, {#CAM.ENABLED}. Filter regex pair ({$MILESTONE.CAM.NAME.MATCHES} / {$MILESTONE.CAM.NAME.NOT_MATCHES}) for include/exclude. M1 adds {#CAM.HW.ID} — the parent hardware GUID — used for host-name dedup so multi-channel encoders don't spawn N hosts on one IP (G2).
A5 — Camera item prototypes that move. icmppingloss, icmppingsec, icmpping are item prototypes on the Camera LLD today, polled at the Site host scope using the camera's IP. M1 moves these to the per-camera-host template. Cleaner host-scoped graphs, cleaner trigger expressions, ICMP runs against the camera host's primary interface instead of being a per-discovered-item config quirk.
A6 — Camera item prototypes that stay. Everything dependent on milestone.cam.raw[{#CAM.ID}] and milestone.cam.ess.raw[{#CAM.ID}] — alarm, channel, enabled, hwmodel, hwname, mac, lastmodified, ESS comm/recording state breakdowns, combined Status. These are all dependent items off batch masters at the Site scope. Moving them to per-camera hosts means N WebSockets / N batch fetches — net loss. Per-camera Zabbix host gets references back via cross-host item lookups in the widget (API::Item()->get() with host=Site,key=milestone.cam.raw[GUID]) — same pattern AP plan uses for fleet-template reads.
A7 — RS LLD with three LLD macros. {#RS.ID} (GUID), {#RS.NAME} (displayName), {#RS.DESCRIPTION}. Item prototypes today: milestone.rs.enabled, milestone.rs.handshake.age, milestone.rs.hostname, milestone.rs.lasthandshake, milestone.rs.version, plus milestone.rs.raw[{#RS.ID}] dependent master and a nodata 15m trigger prototype. All keep their place on the Site host. M1 adds {#RS.HOSTNAME} + {#RS.IP} macros (resolved by the Python helper or a new milestone_rs_state.py) for the host_prototype's agent interface.
A8 — Existing dashboard. Five widgets — item ×4 (site name, mgmt server version, physical memory, API gateway TCP), honeycomb (cameras with alarms, threshold-coloured), itemnavigator (camera list, filterable), itemcard (selected camera details, broadcast-driven), problems (other active problems). The honeycomb is gold — keep it as the centrepiece. What replaces is the itemcard drill-down (becomes the mscamera widget — much richer, sparklines + ESS state breakdown + optional thumbnail) and we add the msserver drill-down which doesn't exist today.
A9 — Combined Status calc. milestone.cam.alarm[{#CAM.ID}] is a CALCULATED item with the formula folding milestone.cam.enabled + icmpping + milestone.cam.ess.comm.type into one of {-1, 0, 1, 2, 3} (Disabled / OK / ESS comm fault but ping OK / Ping down ESS OK / Both). Honeycomb thresholds 1/2/3 = yellow/orange/red. Once ICMP moves to the per-camera host, this calc references a different host. Use last(/Site/icmpping[{#CAM.ADDRESS}]) → last(/Camera-{#CAM.ID}/icmpping). Trip-wire: Zabbix calc items must reference the host explicitly when crossing scopes — gotcha G4 below.
A10 — Macros & valuemaps. 9 host macros ({$MILESTONE.HOST}, .USER, .PASSWORD SECRET, .SCHEME, .CLIENT_ID, .ESS.DELAY, .ESS.STATEGROUP.COMMUNICATION, .ESS.TYPE.COMMUNICATION_OK, plus the camera include/exclude pair). 3 valuemaps (Bool string, Camera status, Service state). All keep their place. M1 adds three new groups: {$MS.AGENT.*} for DVR agent settings, {$MS.CAM.SNMP.*} for vendor SNMP, {$MS.RS.*} for RS-specific overrides.
§B — Predicted gotchas + carry-forward G1–G21 Milestone-specific (G16–G21 added in v1.1) · AP G16–G29 inherited from AP Plan v3.2 §B
Pre-flight list. Validate G1–G15 in M0 and M1 — most are predictions; treat as TODOs not laws. Carry-forwards are settled — accept and apply.
G1 — Camera IP source: /api/rest/v1/cameras doesn't have an address field. The IP belongs to the parent hardware (/api/rest/v1/hardware/{id}). The existing Python helper already joins these — milestone_cameras_state.py — and stamps address + mac onto each camera before writing the snapshot. M1's host_prototype must use this pre-joined address. Calling cameras alone and expecting $.address in the LLD will give blank IPs and the Discovered Cameras host group fills with un-pollable hosts.
G2 — Multi-channel encoders blow up host count. One physical encoder = N "cameras" in Milestone (one per channel). They share one IP, one MAC, one hardware GUID, but get N camera GUIDs. If the host_prototype keys on {#CAM.ID}, you get N Zabbix hosts ICMP-pinging the same IP every minute. Decision: host_prototype keys on {#CAM.HW.ID} (parent hardware GUID), and the Camera LLD does a JS preprocessing dedup pass before emitting LLD entries. Channel-specific items (alarm, ESS state) stay as dependent items at Site scope keyed by {#CAM.ID} — those are free.
G3 — Display-name uniqueness. Milestone allows duplicate camera names across hardware / RS / sites. Zabbix host names must be globally unique. "Front Door" exists 50× across a campus. Decision: host_prototype host: = {#CAM.HW.NAME} if non-empty, falling back to {#CAM.NAME} ({#CAM.HW.IP}). visible_name: stays the human label. Site prefix ({$MILESTONE.SITE.PREFIX}) optional but recommended for multi-site Zabbix instances — defaults empty.
G4 — Calculated-item cross-host references. The existing milestone.cam.alarm calc pulls last(//icmpping[{#CAM.ADDRESS}]) — the // is a same-host shortcut. Once ICMP lives on the per-camera host, the calc on the Site host must do last(/Camera-Host-Name/icmpping) with an explicit host. BUT the host name isn't an LLD-time constant — it's derived from {#CAM.HW.NAME} per G3. Resolution: use last_foreach with a tag filter — e.g. last_foreach(//icmpping?[tag="cam_id" and value="{#CAM.ID}"]). Tag-based foreach is host-agnostic. Works because every per-camera item gets tagged cam_id={#CAM.ID} by the per-camera template. Validate this works in M1's pilot before blanket conversion.
G5 — RS host name vs IP. milestone.rs.hostname field returns FQDN (e.g. rec01.bryant.local). May not resolve from the Zabbix proxy network — the proxy lives on a different VLAN typical. Resolution: the Python milestone_rs_state.py helper (new in M1) does a DNS lookup and stamps both {#RS.IP} (resolved A record) and {#RS.HOSTNAME} (FQDN). host_prototype's agent interface uses IP, useip:'YES'. If DNS fails, the helper marks the RS as dns_unresolved and the LLD filter excludes it (warn-only — Site-host items still poll fine via API).
G6 — Milestone service names vary by version. 2022 R1+: Milestone XProtect Recording Server, Milestone XProtect Event Server, Milestone XProtect Log Server, Milestone XProtect Management Server. Older builds: shorter names without "XProtect" prefix. Some SKUs have additional services (Milestone XProtect Mobile Server, VideoOS Server Logger). Resolution: per-RS template uses an service.discovery LLD filtered by display name regex ^Milestone XProtect.*, then trigger prototypes on the discovered set. Avoids hardcoding names that drift across releases.
G7 — Storage path discovery. Recording archives live at configurable paths (default D:\MediaDatabase). Pull from GET /api/rest/v1/recordingServers/{id}/storageDevices — returns path, maxSize, allocatedSize, plus per-storage retention settings. Per-RS host gets an LLD that calls this endpoint via a JS Script item (the agent reads it back as a JSON master), then emits {#FSPATH} per storage device, then Zabbix Agent's vfs.fs.size[{#FSPATH},*] items poll free-space directly. Trip-wire: the API path is per-RS, but the Zabbix Agent runs on the RS itself — easier path is to call the API from the Site fleet template, then inject the storage-paths array as a host macro on each RS host ({$MS.RS.STORAGE.PATHS}) which the agent template's LLD parses. Pick the cleaner pattern in M1's RS template task.
G8 — License model varies. The existing milestone.license.get hits /licenseOverviewAll — site-level overview. Per-camera license info needs /licenseDetails per RS — and at large sites this is N×100 calls just for license. Decision: mssite widget shows site-level only (sufficient for the operator). Per-camera license drill-down is M5 polish (and only fires on operator click, not on dashboard open).
G9 — Camera GUID casing. Milestone returns lowercase UUIDs. Zabbix LLD macro substitution preserves the case it receives. JSONPath filters in dependent items use $..[?(@.id=='{#CAM.ID}')] — exact-match string comparison. Don't uppercase or normalise the GUID anywhere along the path. The existing template gets this right; the new per-camera template must too — set host: name preserves dash + casing exactly.
G10 — Honeycomb scale ceiling. Zabbix 7.4 honeycomb caps at ~500 items by default (maxlimit). At 2500 cameras the existing all-camera honeycomb truncates silently — operators see a partial view and assume everything is healthy. Resolution: M2's mssite widget renders the honeycomb itself (custom SVG) — no Zabbix limit. Or keep using the native honeycomb but offer per-RS scoping (filter tag: rs_id = X on the items) so each honeycomb is bounded to ≤500 cameras per RS. Cheaper. Pick in M2.
G11 — Vendor SNMP varies wildly. Axis: AXIS-VIDEO-MIB — temperature OID .1.3.6.1.4.1.368.4.1.5.1.0, uptime, status. Hanwha: SAMSUNG-IPCAMERA-MIB — different OID tree, mostly stream-stats focused. Bosch: rich SNMP including PTZ position, codec stats. Hikvision: minimal — just sysDescr / uptime / contact. Decision: M1 ships with one vendor sub-template (Milestone Camera vendor — Axis — most TCS sites are Axis). Other vendors get vendor sub-templates as we encounter them, conditionally linked via sysObjectID regex match in the host_prototype. Cameras with no SNMP get the base template's ICMP only — graceful.
G12 — ICMP scale. 2500 cameras × 1m ICMP = 2500 active checks/min. Default Zabbix proxy StartPollers=5 and StartPingers=1 — that's 60 ICMP/min capacity. Will queue. Resolution: set {$MILESTONE.CAM.PING.INTERVAL} default to 1m at small sites (≤500 cams) and document scaling: 5m at 500–2500, 10m above. Also bump StartPingers on the Milestone proxy. The macro already exists in the current template — keep it. Document the sizing table in M5.
G13 — Camera HTTP probe credentials per vendor. Some support digest auth, some basic, Hikvision has its own ISAPI scheme. Generic web.page.get with macros works for digest+basic. Hikvision/Dahua specific endpoints need a vendor wrapper. Decision: base per-camera template ships ICMP only — no HTTP probe in the base. Vendor sub-templates add the HTTP probe with vendor-correct auth + endpoint. Avoids per-camera credential macros for the common case.
G14 — Trigger duplication risk. The Site host already has milestone.cam.enabled='false' → WARNING and milestone.rs.handshake.age > 300 → HIGH per-prototype triggers. M1 adds per-camera-host triggers (ICMP down, SNMP unreachable). These don't overlap — different signals. But: an "ICMP down" trigger on the camera host AND a "Combined status = 3 Offline" trigger on the Site host will both fire for the same outage. Decision: the Combined-status calc trigger stays the canonical "camera offline" alert. ICMP-down on the camera host is severity information only — it's the input, not the alert. Operators see the calculated alarm; engineers debugging see the ICMP item history.
G15 — ESS WebSocket scope. Tempting to move ESS state to per-camera hosts. Don't. ESS is one WebSocket connection that emits all-camera state. Per-camera scoping = N WebSockets = pointless cost. Keep the existing milestone_ess_state.py daily dump at Site scope; keep the dependent items keyed by {#CAM.ID} there. Per-camera mscamera widget reads them via cross-host API::Item()->get() — same pattern AP plan uses for fleet-template reads (apdetail reads xiq.ap.*[{#SERIAL}] off the XIQ fleet host).
v1.1 additions — surfaced by the Surveillance NOC + Camera Detail + Server Detail mockups
G16 — iDRAC ↔ OS metric duplication. The Server Detail mockup's Server Health rings show CPU%, Memory%, Disk%, and inlet temperature side-by-side. CPU%/Memory% come from the OS-level Zabbix Agent; inlet temp can only come from iDRAC. But iDRAC also exposes cpuUsageIndex (a coarse 0-100 derivative of CPU activity) and a memory-usage estimate — both of which disagree with the OS values by 5–15% because they sample differently. Decision: CPU% and Memory% on the Server Health ring grid are always sourced from the agent (system.cpu.util, vm.memory.utilization) and tagged SourceBadge src="zbx". iDRAC-only metrics (inlet temp, exhaust temp, ambient temp, PSU watts, fan RPM, RAID state) get src="idrac" badge. Never two badges on one cell. If iDRAC is unreachable but agent is up, the temp-ring renders empty with a "iDRAC unreachable" caption — it does not fall back to a guess.
G17 — iDRAC SNMP vs Redfish — pick SNMP for v1, document Redfish for v2. iDRAC exposes both. SNMP is well-trodden (Dell iDRAC-MIB-SMIv2, Zabbix Dell template family ships in box) and easy to LLD across drive bays / PSUs / fans. Redfish is REST/JSON, prettier for nested data (NIC port detail, BMC firmware inventory) but every value needs a JS Script item with HTTPRequest + auth, and credentials live as a host macro per RS host — operationally noisier. v1.1 ships SNMP only. Redfish defers to a v2 enhancement. Risk: some sites disable SNMP on iDRAC for security and only allow Redfish — flag in M0 site survey, treat as a per-site config not a code path.
G18 — iDRAC default credentials are an attack surface — don't ship anything that pre-fills root/calvin. Dell ships iDRACs with the famous default root/calvin. Any dashboard config field that helpfully pre-suggests it makes that mistake operator-visible. Decision: macros {$DELL.IDRAC.SNMP.COMMUNITY} + {$DELL.IDRAC.SNMPV3.SECNAME}+AUTHPASS+PRIVPASS are template-level, default empty, marked type=secret. iDRAC sub-template's items skip-poll if community/secname is unset — the host shows the iDRAC interface but no items. Documentation explicitly: "set credentials per-RS or as a global secret macro on the parent host group; do not hardcode." Mirror the AP plan's SNMPv3 macro shape exactly so ops use one credential pattern across both projects.
G19 — iDRAC LLD: physicalDisk vs virtualDisk numbering drifts across PERC controllers. The Server Detail mockup shows a 24-disk RAID array grid labelled D01..D24. The iDRAC SNMP physicalDiskTable indexes drives by chassis slot — but on PowerEdge R750xs with the front-bay 24-disk backplane, slot indices run 0..23; on R650/R740 with split front+rear cages they're not contiguous; and on PERC H755 controllers in some firmware versions the LLD returns slot strings like "Disk.Bay.0:Enclosure.Internal.0-1:RAID.Integrated.1-1" instead of integers. G20 — ONVIF probing scope and authentication. The Camera Detail mockup shows live RTSP probe results (FPS, bitrate, codec, last-frame-age, ONVIF latency) and surfaces "Stream URL: rtsp://{ip}/onvif/profile1". Reaching that data has three failure modes: (1) ONVIF spec lets vendors choose digest auth (Axis), basic auth (Hanwha older firmwares), or WS-Security UsernameToken (Bosch) — not a universal scheme; (2) the ONVIF service URL itself is discovered via WS-Discovery probes on UDP 3702 with multicast — Zabbix doesn't speak that natively, so we hardcode the path or query Milestone for it; (3) RTSP probe by the Zabbix proxy generates a real video stream connection that some cameras log/audit, generating "stream consumer" events that operators see in Smart Client. Decision: v1.1 does not open RTSP from Zabbix. The "FPS" / "Bitrate" / "Last frame age" cells in mscamera Health source from Milestone (per-camera incoming-stream stats from the API or RS perfcounter VideoOS.Recorder.Service\Stream In\ per-camera instance). ONVIF latency comes from a small HTTP HEAD to the ONVIF endpoint — fast, light, no stream consumed — only when ONVIF is enabled per camera macro. Document this clearly in M3: we monitor "Milestone's view of the stream", not "the stream as a separate consumer". That's the right answer for fleet monitoring; running parallel RTSP at scale would be a denial-of-service against the cameras.
G21 — Camera Wall thumbnail bandwidth at scale. The Surveillance NOC mockup shows a Camera Wall with live thumbnails (8s refresh) — gorgeous demo, but at 1147 cameras × 1 thumbnail / 8s = 143 JPEG fetches/sec from Milestone. Even scoped to "current site" (~200 cameras) it's 25 fetches/sec sustained — Milestone Mobile Server will not love this and operators leaving the dashboard open all day will saturate the API Gateway connection pool. Decision: Camera Wall is per-site only (operator chooses which site, default smallest), opt-in via a tweak in the widget config (default OFF), thumbnails refresh at {$MS.WALL.THUMB.INTERVAL} default 30s not 8s, and the wall pauses thumb refreshes when the dashboard tab loses focus (visibilitychange event). The mockup's "live preview" pane on the Camera Detail page is fine — single camera, not a wall — and the larger sidecar pane refreshes at 8s by design. Document in the M2 closeout that the prettiest mockup feature has the highest scale risk and we shipped a sane default.

Audit triggers across all three new templates plus the iDRAC sub-template. Settle severities to avoid duplicate paging:
Site host (existing): Camera disabled WARNING, RS handshake stale >5m HIGH, ESS snapshot stale WARNING, no-data-for-15m on camera/RS dependent masters AVERAGE. Combined Status calc remains the canonical "camera offline" alert.
Camera host (new): ICMP down 2m INFO only (G14), SNMP unreachable 5m INFO, vendor temperature >60°C AVERAGE, >70°C HIGH, vendor failed-login spike >10/min WARNING, ONVIF auth failure (per G20 — only when ONVIF probe enabled) WARNING.
RS host (Agent-side): Agent unreachable 2m HIGH, Milestone service stopped HIGH, storage volume >90% used AVERAGE, >95% HIGH, perfcounter handle-count growth >20%/24h WARNING (handle leak detector), Recording channels not-recording >0 for 5m AVERAGE, Archive lag >2h AVERAGE.
RS host (iDRAC-side): PSU not OK HIGH, fan not OK AVERAGE, predicted-fail = true on any physicalDisk HIGH (Dell's predictive-fail flag fires before actual failure — earlier warning than smart-error count), virtualDisk degraded AVERAGE, virtualDisk failed HIGH, intrusion sensor tripped HIGH, inlet temp >35°C WARNING, >42°C HIGH (R750xs over-temp threshold), iDRAC unreachable 5m WARNING (informational — not paging level since OS-side may still be healthy), RAID rebuild started INFO (logged but not paged).
Trigger dependency rules (suppress duplicate pages):
(a) Camera ICMP-down (INFO) depends on Site-host Combined-Status (HIGH) — when both fire, only the Combined-Status notifies.
(b) Agent unreachable depends on RS host network-reachable — if the network path is down, agent unreachable is the symptom not the cause.
(c) iDRAC unreachable does not depend on agent unreachable — they're separate paths through the network and we want both signals independently.
(d) physicalDisk predicted-fail does not depend on virtualDisk degraded — predictive-fail fires before the disk actually fails the array; suppressing it once the array is degraded would lose the warning value.
Output: trigger spreadsheet with severities, dependencies, action mappings (page-on-call vs ticket-only vs log-only), and tag set per trigger. Cross-walk against existing on-call rota: anything HIGH wakes someone; anything AVERAGE files a ticket; anything INFO/WARNING is dashboard-only unless it persists for >24h.

Switch-port discovery script — camera ↔ uplink mapping
Build the cross-vendor "discover camera uplink switch + port" external script referenced by M3's Camera Detail sidecar. Pattern follows milestone_rs_state.py: runs from cron every 30m (camera/switch fleets change rarely), outputs JSON keyed by camera MAC. Logic:
(1) Pull the camera-MAC list from the Site fleet template's camera LLD output.
(2) For each switch host in Zabbix tagged role=access-switch, fetch its lldpRemSysName table via API::Item + most-recent history.get. (Switches must already be in Zabbix with the LLDP MIB walked — a project prerequisite documented here.)
(3) Match by chassis MAC: if a camera's MAC appears as an LLDP remote on a switch port, stamp {switch_host_id, switch_port, vlan, poe_class, poe_watts} on the camera in the output.
(4) Cameras with no LLDP match: fall back to FDB-table walk (dot1dTpFdbAddress on the switch via SNMP) to find the port the MAC appeared on. Less reliable (FDB ages out, port may show MAC during a brief packet exchange) but better than nothing.
(5) Camera with neither: stamp uplink: unknown — Camera Detail sidecar shows "Switch not in Zabbix" or "MAC not seen on any switch" with the MAC for manual lookup.
Site fleet template gets a new external item that reads this snapshot and emits per-camera macros ({$CAM.UPLINK.SWITCH}, {$CAM.UPLINK.PORT}, etc.) consumable by mscamera.
Output: milestone_camera_uplink.py + cron entry + documentation. Hand off as part of M5 closeout.
Cross-vendor switch quirks: Cisco IOS LLDP table is cdpCacheTable by default (CDP, not LLDP) unless LLDP is explicitly enabled. Aruba uses standard LLDP-MIB cleanly. Extreme uses standard LLDP. Plan for one Cisco-specific fallback path if TCS has Cisco access switches.
