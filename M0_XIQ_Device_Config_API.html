2 GET /network-policies/{id} — Network Policy

GET /network-policies/{id} Path confirmed from validation checklist — response schema TBD
Returns the full network policy object. The policy ID is obtained from GET /devices/{id} (via network_policy_id or similar field). This call is the bridge between the device and its SSID list — it provides the human-readable policy name and the array of SSID IDs that the policy broadcasts. One GET /ssids/{id} call is then made per SSID ID in that array. Cache this response aggressively (TTL ~5 min) — network policy configuration changes at most a few times per semester.
JSON field (expected) Type Dashboard row / use Panel Status
id int64 Policy ID — used as cache key. Not displayed. — Infer
name string System Info → Network Policy (display name) Overview Infer — mock shows "TCS-Production"
ssid_profile_ids
or ssid_ids / ssids		 int64[] Drives the fan-out to GET /ssids/{id} — one call per ID Wireless tab Infer — array of SSID profile IDs; validate field name
description string Optional — tooltip or hover detail on the Network Policy row. Not a required display field. Overview TBD
If /devices/{id} already returns network_policy_name as a string, skip this endpoint for the System Info Network Policy display row — you already have the name. Only call /network-policies/{id} if the device response gives an ID only, or to obtain the SSID ID list. Validate during M0 step 5 curl.
Fan-out limit: TCS-Production uses 3 SSIDs (TCS-Wireless, TCS-Guest-Registration, TCS-VIPGuest). Never make more than the confirmed count of SSID calls. If the SSID ID array is unexpectedly long (e.g. 20+ SSIDs from a shared policy), cap the fan-out and log a warning — a misconfigured XIQ policy could spike your per-load call count well beyond the 8-call budget.

GET /ssids/{id} Path inferred — VALIDATE exact path and response schema
Returns the configuration profile for one SSID. Called once per SSID ID returned by GET /network-policies/{id}. For the BHS pilot this is 3 calls. Clients column — SSID Broadcast table

The Clients column in the SSID Broadcast table (count per SSID) is not returned by /ssids/{id}. Source options:

NAC Role column — SSID Broadcast table

The NAC Role column is PF-only. XIQ has no concept of PF roles. Source is the PF connection profile name associated with each SSID. Implementation: look up the PF realm or connection_profile configured for each SSID name, or derive from the active session role distribution in locationlog.

6 PacketFence Overlap Analysis

Key principle: XIQ and PacketFence provide different views of the same facts. XIQ describes what is configured (policy intent). PacketFence describes what happened per client per session (operational reality). These are complementary, not redundant. No field should be dropped from either source because the other covers it.
Field XIQ provides PF provides Resolution — which wins where
SSID name XIQ /ssids/{id} → ssid_name — configured SSID name PF locationlog.ssid — SSID the client actually connected to at session time XIQ wins for SSID Broadcast table (config definition). PF wins for Clients tab SSID column (per-client session truth). The two should match — if they don't for a session, that's an anomaly worth flagging.
VLAN XIQ → configured default VLAN for the SSID PF locationlog.vlan — VLAN PF actually assigned to each client (may vary by role) XIQ wins for SSID Broadcast table VLAN column (shows the policy intent). PF wins for Clients tab VLAN column (shows per-client assignment). For role-based VLANs, XIQ will show only the default; PF shows the actual assigned VLAN — both are needed.
Auth method XIQ → configured security type per SSID (e.g. WPA2-Enterprise, Captive Portal) PF locationlog.connection_type — EAP method actually used per client (Wireless-802.11-EAP, PEAP, EAP-TLS, MAC-Auth) XIQ wins for SSID Broadcast table Auth column (config-level auth type). PF wins for Clients tab Auth column (per-client actual EAP method). Both are needed — they answer different questions.
Client count (per AP) XIQ /clients?device_id=X → current connected clients, point-in-time PF locationlog active sessions (end_time = 0000...) — current NAC-registered clients XIQ count = RF-associated clients. PF count = NAC-registered clients. They will differ: a client can be RF-associated but not yet NAC-registered (or vice versa during posture check). Show both in the Overview Issues panel as separate metrics if counts diverge significantly.
Client MAC address XIQ /clients → mac_address PF locationlog.mac and nodes.mac MAC is the JOIN key — use it to produce the enriched client row. Neither source "wins"; both are needed for the join. Normalize to uppercase colon-separated format before joining.
Client hostname XIQ /clients → hostname PF node.computername PF wins — PF uses the DHCP-registered hostname which is more reliable for inventory (e.g. "BHS-CHROME-1142"). XIQ hostname may be mDNS-discovered and can be empty. Use PF as primary, XIQ as fallback.
Encryption XIQ /ssids/{id} → configured cipher type Not in PF XIQ only — no overlap. SSID Broadcast table Encryption column is XIQ-only.
Band / radio XIQ /clients → radio_index / band Not in PF locationlog XIQ only — PF does not record radio band. Band column in Clients tab comes entirely from XIQ.
RSSI / signal strength XIQ /clients → rssi Not in PF XIQ only — PF has no RSSI. RSSI bar in Clients tab is XIQ-only.
NAC role / posture Not in XIQ PF node.status, node.category PF only — XIQ has no NAC role or posture concept. All role/posture data in Clients tab is PF-only.

7 Caching Strategy — Config vs. Live Endpoints

All config-class data changes on operator action only — not continuously. Cache aggressively to protect the 7,500 req/hr quota. Live session data (clients, radio stats) must not be cached.

/devices/{id} 60 s Device status and client count change frequently — short TTL.
/clients (filtered) No cache — live Real-time client sessions. Fetch every dashboard load.
/devices/{id}/wifi-if-stats No cache — live Current RF metrics snapshot. Fetch every dashboard load.
/network-policies/{id} 5 min Policy config changes only on deliberate network team action.
/ssids/{id} ×N 5 min Same as policy — cache each SSID by its ID as the key.
/devices/{id}/location 10 min Changes only when AP is physically moved. Very slow-changing. 
// XIQClient.php — APCu cache key conventions
// Namespace all keys under "xiq:" to avoid collision with other APCu users

$device_cache_key  = "xiq:device:{$xiq_device_id}";        // TTL: 60s
$policy_cache_key  = "xiq:policy:{$network_policy_id}";     // TTL: 300s
$ssid_cache_key    = "xiq:ssid:{$ssid_id}";                  // TTL: 300s (per SSID)
$location_cache_key= "xiq:location:{$xiq_device_id}";        // TTL: 600s

// Live endpoints — never cached:
// GET /clients?device_id=X
// GET /devices/{id}/wifi-if-stats

// Cache the resolved XIQ device ID separately (almost never changes):
$id_cache_key      = "xiq:id:host:{$zabbix_host_id}";        // TTL: 3600s (1h)
APCu is per-PHP-worker-process. If the Zabbix web server spawns multiple PHP-FPM workers (typical default: 5–10), each worker will have its own APCu segment. A cache set in worker A is not visible to worker B. This means up to 10× your expected call count until each worker primes its cache independently. This is the expected behavior — it does not indicate a bug. The 7,500 req/hr quota with 10 workers and 10 operator sessions is still within budget, but monitor RateLimit-Remaining headers in production.

8 M0 Validation — Additional Curl Commands for Config Endpoints

These supplement the validation checklist in M0_XIQ_Wireless_API_Reference §9. Run after obtaining $TOKEN and $DEVICE_ID from Steps 1–2 of that checklist.

# Step A — Full /devices/{id} response — grep for config/NTP fields
curl -s https://api.extremecloudiq.com/devices/$DEVICE_ID \
  -H 'Authorization: Bearer $TOKEN' | python3 -m json.tool | grep -iE \
  "ntp|config|policy|software|version|status|updated|time"
# VALIDATE: ntp_server presence, config_status field name, software_version,
# network_policy_id vs network_policy_name, last_updated unit (check if value
# looks like unix seconds (~1.7e9) or unix ms (~1.7e12))

# Step B — Extract network_policy_id from device response
POLICY_ID=$(curl -s https://api.extremecloudiq.com/devices/$DEVICE_ID \
  -H 'Authorization: Bearer $TOKEN' | python3 -c \
  "import sys,json; d=json.load(sys.stdin); print(d.get('network_policy_id','FIELD_NOT_FOUND'))")
echo "Policy ID: $POLICY_ID"

# Step C — Network policy (validate name field + SSID ID array field name)
curl -s https://api.extremecloudiq.com/network-policies/$POLICY_ID \
  -H 'Authorization: Bearer $TOKEN' | python3 -m json.tool
# VALIDATE: response is an object (not paginated list), field names for:
# policy name (expected: "name"), SSID ID array (expected: "ssid_profile_ids" or "ssid_ids")

# Step D — Single SSID detail (use first ID from policy response)
curl -s https://api.extremecloudiq.com/ssids/$SSID_ID_1 \
  -H 'Authorization: Bearer $TOKEN' | python3 -m json.tool
# VALIDATE: ssid_name / name, vlan_id, security_type enum values,
# encryption_method, band fields (enable_24ghz / enable_5ghz or single band enum)

# Step E — If /ssids/{id} returns 404, try alternate paths:
curl -s https://api.extremecloudiq.com/ssid-profiles/$SSID_ID_1 \
  -H 'Authorization: Bearer $TOKEN'
curl -s https://api.extremecloudiq.com/wifi-networks/$SSID_ID_1 \
  -H 'Authorization: Bearer $TOKEN'

# Step F — Location (validate field names, coordinate system)
curl -s https://api.extremecloudiq.com/devices/$DEVICE_ID/location \
  -H 'Authorization: Bearer $TOKEN' | python3 -m json.tool
# VALIDATE: building_name / building.name field path, floor_name, x/y vs lat/lng,
# presence of floor_plan_image_id or similar, installation_date field

# Step G — Check if floor plan image requires auth (do NOT hardcode token in img src)
# If location response contains an image URL, try fetching it without auth first:
curl -I "$FLOOR_PLAN_URL"
# 200 without auth = public URL, safe for <img src="">
