§A — Two-Template Zabbix Architecture New in v3
How XIQ data lives in Zabbix · what the widget actually queries · why this matters for rate-limiting and history

The biggest shift since v2 is that most XIQ data is now polled by Zabbix, not by the widget. Two templates work together: a fleet template polls XIQ centrally, and a per-AP template polls each AP's SNMP agent. The widget reads from both via API::Item()->get() and only calls XIQ live for things that change second-by-second (current-bucket radio stats, currently-associated clients). This drops live XIQ traffic per page load from ~7 calls to ~3, gives Zabbix history for fields XIQ doesn't expose history for, and lets us alert on XIQ-sourced metrics (config mismatch, disconnected APs) using normal Zabbix triggers.

FLEET TEMPLATE — Extreme XIQ APs by API
Master Script item runs every 5 min on a single fleet host: paginates GET /devices?views=FULL, returns one flat JSON array. Dependent items extract per-AP fields by serial. LLD with host prototype auto-creates one Zabbix host per AP under Discovered APs/Wireless/{location}, links the per-AP template, stamps host macros and tags.

Cached items (per AP, JSONPath-extracted):
xiq.ap.xiqid[*] · xiq.ap.model[*] · xiq.ap.mac[*] · xiq.ap.ip[*] · xiq.ap.version[*] · xiq.ap.policy[*] · xiq.ap.connected[*] · xiq.ap.lastconnect[*] · xiq.ap.uptime[*] · xiq.ap.clients[*] · xiq.ap.configmismatch[*] · xiq.ap.adminstate[*] · xiq.ap.locationid[*]
PER-AP TEMPLATE — Extreme AP via SNMPv3
Auto-linked at host prototype creation. SNMPv3 agent against the AP's Mgt0 IP. Polls Aerohive enterprise MIB .26928 + standard IF-MIB. Two LLD rules: radio interfaces (wifi0/wifi1) and SSID subinterfaces.

Scalar items:
extremeap.cpu.util · extremeap.mem.util · extremeap.serial.0 · extremeap.firmware.0 · extremeap.noise.wifi0 · extremeap.noise.wifi1 · net.if.in[ifHCInOctets.10] · net.if.out[ifHCOutOctets.10] · net.if.status[ifOperStatus.10] · system.uptime

LLD prototypes:
extremeap.channel[*] · extremeap.txpower[*] · extremeap.chanwidth[*] · extremeap.ssid.name[*] · net.if.errors[ifInErrors.*]
What this means for widget code: WidgetView::doAction() first calls API::Item()->get() with the AP's hostid and gets every cached value back in one batch. Live XIQ calls reduce to: /devices/{id}/interfaces/wifi (radio current values, 15-min window — G6), /clients/active?deviceIds={id}&views=FULL (associated clients — G4, G5), and the d360 sparkline endpoints. Live PF calls: /nodes/search, /radius_audit_log. Everything else — model, MAC, firmware, network policy name, location, SSID names, uplink rate, CPU%, memory%, noise floor — comes from Zabbix items that already have history.
§B — Live-Validation Gotchas Updated in v3.2 — 33 total · G16–G33 new from M1 closeout
33 issues · G1–G15 from M0+v3.1 · G16–G33 new in v3.2 from M1 iteration cycle (Zabbix 7.4 widget-framework + PHP 8.0 + smoke-test findings) · full detail in M0_Closeout_Reference.html §2 + M0_PacketFence_API_Live_Findings.html
G1 — PF radius_audit_logs/search filter is single starts_with on nas_ip_address, not the dual-IP op:or that v3 specified. The AP is the NAS in 802.1X; nas_ip_address is vendor-independent (called_station_id format varies). Updated in v3.1.
G2 — PF time-range operator is greater_than, NOT greater_than_equals. PF 15 returns 422 "op is not valid" for the latter. PFClient compensates with $from − 1 to preserve inclusive boundary. Reversed in v3.1 from v3.
G3 — Each AP has TWO MACs. 74:26:AC:2C:A4:90 (eth0, used by SNMP + PF) vs BC:F3:10:04:C0:40 (wireless base, used by XIQ). Per-radio BSSIDs derive from the latter. PFClient uses switch_ip not MAC prefix to sidestep this entirely.
G4 — XIQ query params are camelCase plural: deviceIds, locationIds, vlans, ssids. Sending device_id silently returns fleet-wide data. XIQClient line 949 uses deviceIds.
G5 — XIQ /clients/active default views emits 12 fields. Append views=FULL for the 50+ field schema (rssi, snr, client_health, locations[]). XIQClient line 950 sets it.
G6 — XIQ radio stats need a 15-min startTime/endTime window. 5-min returns empty; spec says optional but server returns 400. WIFI_STATS_WINDOW_SEC = 900.
G7 — XIQ /devices/{id}/ssid/status is keyed by SSID id, not name. JOIN against /network-policies/{policyId}/ssids for names + security.
G8 — XIQ /d360/wireless/interfaces-graph channel param is a radio selector (WIFI0 / WIFI1 / WIFI2), NOT an 802.11 channel number. New in v3.1.
G9 — XIQ d360 graph arrays come unsorted. Sort by timestamp ascending before passing to the JS sparkline renderer.
G10 — Pilot AP runs dual-5G (channels 40 and 149). Don't assume radio 0 = 2.4 GHz. Derive band from the XIQ frequency field or from mac_protocol string ("802.11ax-5g"), never from radio_index.
G11 — XIQ /clients/active is the source of truth for "currently connected". PF locationlog "active" sentinel can lag — don't surface PF-only sessions as currently connected; use PF for enrichment only.
G12 — PF nodes/search doesn't accept op:in. Use op:or wrapping N op:equals clauses for batch MAC lookup. PFClient::nodeSearchBody() uses this pattern.
G13 — PF Reject reason field is verbose RADIUS module output. Auth Failures table needs a regex summary ("wrong password" / "user not found" / "EAP timeout") with full text on hover.
G14 — PF switch_ip exists on the locationlog table but NOT on the node table. Filtering /nodes/search by switch_ip returns 422 "switch_ip is an invalid field". Connected Clients flow needs two calls: locationlog (filter by switch_ip → open-session MAC list) then nodes/search (filter by those MACs). Encapsulated in PFClient::getActiveClientsForAp(). New in v3.1.
G15 — PF radius_audit_logs/search returns HTTP 404 with body {"errors":[{"message":"entries not found"}]} when a valid query matches zero rows. Other PF endpoints (locationlogs, nodes) return 200 + empty items[] for the same case. PFClient::radiusAuditLog() translates the 404 to empty-success. New in v3.1.
Zabbix 7.4 widget framework — G16–G29 new in v3.2
G16 — Module namespaces must be globally unique across ALL installed Zabbix modules, not just within the project. Two modules sharing namespace → PHP autoload binds the FQN to whichever loaded first; the loser's classes shadow at runtime. Symptom triplet: module visible in admin Modules list but missing from dashboard widget picker AND enabling it breaks an unrelated existing widget. Pick a deployment-specific namespace prefix from the start (we used TcsApDetail after APDetail collided with another widget). New in v3.2.
G17 — Zabbix 7.4 manifests need "type": "widget" at top level. Without it the module appears in Administration → Modules but never registers as a dashboard widget type — picker doesn't list it. New in v3.2.
G18 — Zabbix 7.4 manifests need a top-level "widget": {"js_class": "...", "size": {"width": 24, "height": 8}} block. js_class must match the JS class name (e.g. CWidgetAPDetail). Without this block, picker registration silently drops the widget. New in v3.2.
G19 — Asset paths go under a top-level "assets": {"js": [...], "css": [...]} block, with paths relative to assets/js/ and assets/css/ respectively. The flatter form ("js": ["assets/js/class.widget.js"] at top level) works for some Zabbix module loaders but breaks the dashboard widget loader. New in v3.2.
G20 — Widget.php extends Zabbix\Core\CWidget, NOT CModule. CModule registers as a generic non-widget module (visible in admin Modules list, never in widget picker). The M0 scaffold had extends CModule from day one — picker never worked until M1 found this. New in v3.2.
G21 — Deployment is PHP 8.0.30, not 8.1+ as the project prompt previously claimed. Avoid all 8.1-only constructs: readonly properties (parses as class name on 8.0 and chokes — XIQClient hit this on 4 properties), array_is_list() (doesn't exist), enums, never return type, new in initializers, intersection types. Polyfills for missing 8.1 standard-library functions must be declared inside the calling namespace — PHP function lookup is namespace-first then global, so a polyfill in Modules\Foo\Actions doesn't help calls from Modules\Foo\Includes. New in v3.2.
G22 — Host fields use CWidgetFieldMultiSelectHost with field name hostids (plural), NOT CWidgetFieldHost with hostid (singular). The singular variant does not appear in $data['fields'] in 7.4 form rendering, breaking widget.edit.php. Multi-select handles single-host intent fine; broadcast-receiver wire type stays _hostid singular. New in v3.2.
G23 — CWidgetFieldIntegerBox in 7.4 only exposes setDefault() for configuration — there is no setMin() / setMax(). Calling them throws Call to undefined method mid-addFields() chain, which kills every later field declaration in the same chain (visible as $data['fields']['xxx'] missing log entries from defensive code). Same caveat applies to FLAG_LABEL_ASTERISK as a class constant — use FLAG_NOT_EMPTY alone. New in v3.2.
G24 — refresh_rate is a reserved system field name in Zabbix 7.x. The framework injects a refresh-interval picker into every widget edit dialog automatically; declaring your own field with that name causes it to be stripped out of $data['fields']. Workaround in xiq_ap_status: use rf_rate. Better: rely on the framework default and don't declare anything. New in v3.2.
G25 — View output must go through (new CWidgetView($data))->addItem($body)->show(). Direct HTML echoes (<?= ?> or <div> markup outside the CWidgetView tree) bypass the response pipeline that JSON-wraps for the dashboard's AJAX layer. Symptom: SyntaxError: Unexpected token '<', "\n<div class"... is not valid JSON. Build the body with CDiv / CTag / CTable objects, hand the tree to CWidgetView::show(). New in v3.2.
G26 — Custom top-level keys in CControllerResponseData get stripped by Zabbix's widget AJAX layer; only name, data, user, and a few others survive to the view. Pack everything operator-visible under a single "data" key on the controller side and unpack as $data['data'] in the view. Same pattern xiq_ap_status uses. New in v3.2.
G27 — JS lifecycle overrides (onInitialize, onActivate, onDeactivate, setContents) MUST call super.<method>() first. The CWidget base class's onInitialize wires internal state that enterWidgetEditing / leaveWidgetEditing later toggle — skip super and the next dialog close throws TypeError: Cannot set properties of undefined (setting 'disabled'). New in v3.2.
G28 — DOM rendering and event binding go in setContents(response), NOT onActivate(). setContents is the canonical hook for AJAX content arrival; it runs every refresh. onReady() is not a real CWidget hook — was guessed from another framework. Use this._body (set up by super.setContents) for DOM access. New in v3.2.
G29 — Read saved widget field values via $this->fields_values['name'] ?? $default, NOT $this->getInput('name'). getInput reads from the HTTP request and warns "Undefined array key" on first render before any save POST. fields_values is the framework-populated property where saved widget config + broadcast values both land. Only use getInput for things that genuinely arrive via the request (name is set by Zabbix automatically). New in v3.2.
XIQClient — G30–G33 new in v3.2 · M2/M3/M5 work-items
G30 — Actual XIQClient method names diverge from M0 Closeout §6 reference. Real: getInterfacesGraph(int $deviceId, string $radio, int $from, int $to, string $source = '') — not getWirelessGraph; getClients(int $deviceId) is the single roster method that replaces the trio getActiveClients / getClientGrid / getClientGraph. getDeviceIssues and getDeviceAlarms do not exist on current XIQClient — Connectivity Issues panel + secondary Recent Events feed in M2 need either new wrappers or alternative sources (Zabbix triggers + PF audit log already merge cleanly per M5). Update §6 reference to match implementation. New in v3.2.
G31 — XIQClient::getClients() sends a hard-coded limit=500 to /clients/active. XIQ rejects with HTTP 400 "listActiveClients.limit: must be less than or equal to 100". Cap at 100 and add pagination if any AP genuinely has >100 clients (BHS pilot has ~9, single page is plenty for now). M2 fix-on-arrival. New in v3.2.
G32 — XIQClient::getLocation() calls /locations/{id} with HTTP GET; XIQ replies HTTP 400 "Request method 'GET' not supported". The endpoint is POST. Blocks M5 sidecar (floor plan + coordinates). M5 fix-on-arrival. New in v3.2.
G33 — XIQClient::getPolicySsids() fans out internally to per-SSID lookups at /ssids/{id} which return HTTP 403 AUTH_ACCESS_DENIED with the current {$XIQ_API_TOKEN} scope. Two fixes possible: (a) skip the per-SSID fan-out and read SSID summaries from the policy response directly (faster and avoids the scope issue entirely), or (b) regenerate the XIQ token with view_ssid scope. Affects M3 SSID Broadcast table. M3 fix-on-arrival. New in v3.2.
