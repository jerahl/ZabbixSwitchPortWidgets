§A — Two-Template Zabbix Architecture New in v3
How XIQ data lives in Zabbix · what the widget actually queries · why this matters for rate-limiting and history

The biggest shift since v2 is that most XIQ data is now polled by Zabbix, not by the widget. Two templates work together: a fleet template polls XIQ centrally, and a per-AP template polls each AP's SNMP agent. The widget reads from both via API::Item()->get() and only calls XIQ live for things that change second-by-second (current-bucket radio stats, currently-associated clients). This drops live XIQ traffic per page load from ~7 calls to ~3, gives Zabbix history for fields XIQ doesn't expose history for, and lets us alert on XIQ-sourced metrics (config mismatch, disconnected APs) using normal Zabbix triggers.

FLEET TEMPLATE — Extreme XIQ APs by API
Master Script item runs every 5 min on a single fleet host: paginates GET /devices?views=FULL, returns one flat JSON array. Dependent items extract per-AP fields by serial. LLD with host prototype auto-creates one Zabbix host per AP under Discovered APs/Wireless/{location}, links the per-AP template, stamps host macros and tags.

Cached items (per AP, JSONPath-extracted):
xiq.ap.xiqid[*] · xiq.ap.model[*] · xiq.ap.mac[*] · xiq.ap.ip[*] · xiq.ap.version[*] · xiq.ap.policy[*] · xiq.ap.connected[*] · xiq.ap.lastconnect[*] · xiq.ap.uptime[*] · xiq.ap.clients[*] · xiq.ap.configmismatch[*] · xiq.ap.adminstate[*] · xiq.ap.locationid[*]
PER-AP TEMPLATE — Extreme AP via SNMPv3
Auto-linked at host prototype creation. SNMPv3 agent against the AP's Mgt0 IP. Polls Aerohive enterprise MIB .26928 + standard IF-MIB. Two LLD rules: radio interfaces (wifi0/wifi1) and SSID subinterfaces.

Scalar items:
extremeap.cpu.util · extremeap.mem.util · extremeap.serial.0 · extremeap.firmware.0 · extremeap.noise.wifi0 · extremeap.noise.wifi1 · net.if.in[ifHCInOctets.10] · net.if.out[ifHCOutOctets.10] · net.if.status[ifOperStatus.10] · system.uptime

LLD prototypes:
extremeap.channel[*] · extremeap.txpower[*] · extremeap.chanwidth[*] · extremeap.ssid.name[*] · net.if.errors[ifInErrors.*]
What this means for widget code: WidgetView::doAction() first calls API::Item()->get() with the AP's hostid and gets every cached value back in one batch. Live XIQ calls reduce to: /devices/{id}/interfaces/wifi (radio current values, 15-min window — G6), /clients/active?deviceIds={id}&views=FULL (associated clients — G4, G5), and the d360 sparkline endpoints. Live PF calls: /nodes/search, /radius_audit_log. Everything else — model, MAC, firmware, network policy name, location, SSID names, uplink rate, CPU%, memory%, noise floor — comes from Zabbix items that already have history.
§B — Live-Validation Gotchas Updated in v3.1
15 issues · G1, G2, radius endpoint corrected from PFClient implementation · G14, G15 new · full detail in M0_Closeout_Reference.html §2 + M0_PacketFence_API_Live_Findings.html
G1 — PF radius_audit_logs/search filter is single starts_with on nas_ip_address, not the dual-IP op:or that v3 specified. The AP is the NAS in 802.1X; nas_ip_address is vendor-independent (called_station_id format varies). Updated in v3.1.
G2 — PF time-range operator is greater_than, NOT greater_than_equals. PF 15 returns 422 "op is not valid" for the latter. PFClient compensates with $from − 1 to preserve inclusive boundary. Reversed in v3.1 from v3.
G3 — Each AP has TWO MACs. 74:26:AC:2C:A4:90 (eth0, used by SNMP + PF) vs BC:F3:10:04:C0:40 (wireless base, used by XIQ). Per-radio BSSIDs derive from the latter. PFClient uses switch_ip not MAC prefix to sidestep this entirely.
G4 — XIQ query params are camelCase plural: deviceIds, locationIds, vlans, ssids. Sending device_id silently returns fleet-wide data. XIQClient line 949 uses deviceIds.
G5 — XIQ /clients/active default views emits 12 fields. Append views=FULL for the 50+ field schema (rssi, snr, client_health, locations[]). XIQClient line 950 sets it.
G6 — XIQ radio stats need a 15-min startTime/endTime window. 5-min returns empty; spec says optional but server returns 400. WIFI_STATS_WINDOW_SEC = 900.
G7 — XIQ /devices/{id}/ssid/status is keyed by SSID id, not name. JOIN against /network-policies/{policyId}/ssids for names + security.
G8 — XIQ /d360/wireless/interfaces-graph channel param is a radio selector (WIFI0 / WIFI1 / WIFI2), NOT an 802.11 channel number. New in v3.1.
G9 — XIQ d360 graph arrays come unsorted. Sort by timestamp ascending before passing to the JS sparkline renderer.
G10 — Pilot AP runs dual-5G (channels 40 and 149). Don't assume radio 0 = 2.4 GHz. Derive band from the XIQ frequency field or from mac_protocol string ("802.11ax-5g"), never from radio_index.
G11 — XIQ /clients/active is the source of truth for "currently connected". PF locationlog "active" sentinel can lag — don't surface PF-only sessions as currently connected; use PF for enrichment only.
G12 — PF nodes/search doesn't accept op:in. Use op:or wrapping N op:equals clauses for batch MAC lookup. PFClient::nodeSearchBody() uses this pattern.
G13 — PF Reject reason field is verbose RADIUS module output. Auth Failures table needs a regex summary ("wrong password" / "user not found" / "EAP timeout") with full text on hover.
G14 — PF switch_ip exists on the locationlog table but NOT on the node table. Filtering /nodes/search by switch_ip returns 422 "switch_ip is an invalid field". Connected Clients flow needs two calls: locationlog (filter by switch_ip → open-session MAC list) then nodes/search (filter by those MACs). Encapsulated in PFClient::getActiveClientsForAp(). New in v3.1.
G15 — PF radius_audit_logs/search returns HTTP 404 with body {"errors":[{"message":"entries not found"}]} when a valid query matches zero rows. Other PF endpoints (locationlogs, nodes) return 200 + empty items[] for the same case. PFClient::radiusAuditLog() translates the 404 to empty-success. New in v3.1.

M4 decision: Connected Clients join strategy (XIQ + PF, XIQ-only, or PF-only)
Three viable patterns:
(a) XIQ + PF join (full enrichment): XIQClient::getClients() for live RSSI/SNR/data rate + PFClient::getActiveClientsForAp() for user/role/posture/OS, joined on mac_raw ↔ session.mac. Two live API calls per page load. Recommended for the Clients tab.
(b) XIQ-only: just getClients(). No NAC enrichment — operator can't see role/posture. Cheaper but loses key value.
(c) PF-only: just getActiveClientsForAp(). No live RSSI/SNR — can't see signal quality. Useful as XIQ-degraded fallback.
Decide before M4 implementation begins. M3 SSID Broadcast already needs PF data (NAC role per SSID), so PFClient is in the call path either way — option (a) adds one row-level join, not a whole new client.
