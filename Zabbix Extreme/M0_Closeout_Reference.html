2 Critical implementation gotchas

These are the things that surprised us during validation and will bite during implementation if not handled. Each was found empirically; assume that anyone reading project docs without seeing the validation transcripts will not know about them.

G1 — PF Rejects do not populate switch_ip_address. On radius_audit_logs, Accept records have switch_ip_address = "172.16.97.59" but Reject records have switch_ip_address = "N/A" (literal string) and the AP IP appears in nas_ip_address instead. The Auth Failures table query must op:or both fields; v4's single-field filter returned 41 records with 0 Rejects despite Rejects existing on this AP. 
// CORRECT — captures Accepts AND Rejects for one AP
{
  "query": {
    "op": "or",
    "values": [
      { "field": "switch_ip_address", "op": "equals", "value": "172.16.97.59" },
      { "field": "nas_ip_address",    "op": "equals", "value": "172.16.97.59" }
    ]
  }
}
G2 — PF time-range operator is greater_than_equals (no "or", with trailing s). The project's earlier M0_PacketFence_Historical_API.html doc shows greater_than_or_equals — this is wrong and PF 15 returns 422 for it. Three operators work in v5 testing: greater_than, greater_than_equals, not_equals. between rejects the value-shape [from, to]; gte/gt/after all 422.
G3 — Each AP has TWO MACs. 74:26:AC:2C:A4:90 is the eth0 management MAC (used by Zabbix SNMP, used as PF switch_mac for the AP's wired uplink record). BC:F3:10:04:C0:40 is the wireless base MAC reported by XIQ /devices/{id}.mac_address; per-radio BSSIDs derive from it (…C054, …C064) with per-SSID variants (…C056, …C066). Always filter PF logs by switch_ip_address not by MAC prefix — sidesteps the dual-MAC problem and works for both Accept and Reject (with G1 applied).
G4 — XIQ /clients/active deviceIds param is camelCase plural; sending device_id silently returns fleet-wide. XIQ ignores unknown query params without error. v3+v4 lost an entire iteration to this. The same camelCase-plural pattern applies to locationIds, vlans, userProfileNames, ssids, clientOsNames, userNames, clientConnectionTypes.
G5 — XIQ /clients/active default views emits only 12 fields. The full XiqClient schema has 50+ fields including rssi, snr, channel, mac_protocol, client_health, radio_health, connection_duration, locations[]. To get them, append views=FULL. Enum: BASIC | FULL | STATUS | DETAIL | LOCATION | METRICS | IOT | THREAD.
G6 — XIQ radio stats require startTime and endTime even though the OpenAPI spec marks them optional. The snapshot form returns 400 "Required request parameter 'startTime' for method parameter type long is not present". A 5-minute window also returns [] (empty) — XIQ buckets at ~10-minute intervals so a window shorter than the bucket misses data. Use a 15-minute window for "current" radio values.
G7 — XIQ /devices/{id}/ssid/status returns a map keyed by SSID id, not name. To populate the SSID Broadcast table you must JOIN this map against /network-policies/{policyId}/ssids (which gives id → name + security). Both calls are cheap and ~5-min cacheable.
G8 — XIQ /d360/wireless/interfaces-graph channel= param is a radio selector despite its description. Description says "channel utilization type" but the enum is ['WIFI0','WIFI1','WIFI2']. The actual metric selector is the optional source= with enum ['CHANNEL_UTILIZATION','CONNECTED_CLIENTS'] (default = both). Issue one call per radio.
G9 — XIQ time-series timestamps are NOT chronologically ordered in the response. Sort by timestamp ascending in PHP before passing to the JS sparkline renderer. See xiq_wireless_graph_wifi0.json: 7 samples in random temporal order.
G10 — The pilot AP runs dual-5G mode, not 2.4 GHz + 5 GHz. Both wifi0 (channel 40) and wifi1 (channel 149) report frequency: "5G" with the same radio_profile_name: "TCS-NEW-5GHZ". The Wireless tab must enumerate radio cards from the API response — never hardcode card count or band labels. Other AP_305C deployments may run 2.4 + 5 + 5 (tri-radio).
G11 — XIQ "currently connected" ≠ PF "active session". XIQ /clients/active returned 3 clients on this AP; PF locationlog returned 9 active (end_time='0000-…'). PF's active sentinel is sticky and doesn't always clear when a client drops without a clean disassociate. Use XIQ as the source of truth for the Connected Clients table; PF supplies enrichment per MAC. Don't show PF-only "active" sessions as currently connected — they're stale.
G12 — PF nodes/search doesn't accept op:in. Use op:or wrapping N op:equals clauses for batch MAC lookup. The per-batch cost is one HTTP call regardless of MAC count, and PF accepts hundreds of OR-clauses per query based on observed limits (request size, not clause count, is the constraint).
G13 — The PF reason field on Reject records is verbose RADIUS module output. Sample: "chrooted_mschap: Program returned code (3) and output 'NT Error: code: 3221225578, message: …value provided as the current password is not correct…'". The Auth Failures table should display a short summary (extract eap_type + a regex-pulled summary like "wrong password" / "user not found" / "EAP timeout") with the full reason on hover/expand.

All endpoints below confirmed live on api.extremecloudiq.com against this AP. The OpenAPI spec lists cloudapi.extremecloudiq.com/xiq/v1 as canonical but api.extremecloudiq.com is the working public alias.

4 PacketFence confirmed endpoint contracts

PF base: https://<PF_HOST>:9999/api/v1/. Self-signed cert — disable verification in PHP curl (CURLOPT_SSL_VERIFYPEER=0). Auth via POST /login; bearer token valid 1 hour by default.

Query grammar (canonical)

Operators confirmed working in PF 15: and, or, equals, not_equals, greater_than, greater_than_equals, starts_with (some endpoints).
Operators that 422: in, gte, gt, greater_than_or_equal, greater_than_or_equals, greater_or_equal, after, between.

Active session enumeration

POST /api/v1/locationlogs/search current
Query body filters by switch_ip = AP_MGT_IP and end_time = '0000-00-00 00:00:00' sentinel. Returns currently-associated MACs per the AP's RADIUS history. Use as the input MAC list for the nodes/search batch lookup, not as the connected-clients list directly (G11). Pilot result: 9 MACs.

  "limit": 200,

5 Cross-source join recipes

Connected Clients table (Clients tab)

XIQ is the source of truth (G11); PF enriches per MAC.

// 1. Get currently-connected clients on this AP
$xiq = $xiqClient->getActiveClients($apXiqId);
// $xiq[i] = { mac_address, hostname, ip_address, ssid, vlan, rssi, snr, … }

// 2. Collect MACs in PF colon-form (XIQ returns no-delimiter)
$macs = array_map(fn($c) => PFClient::macColonize($c['mac_address']), $xiq);

// 3. Batch enrich via PF nodes/search op:or-of-equals
$pf = $pfClient->nodesByMac($macs);   // keyed by lowercase MAC

// 4. Merge per-row (XIQ left, PF right)
$rows = [];
foreach ($xiq as $c) {
    $mac = PFClient::macColonize($c['mac_address']);
    $node = $pf[$mac] ?? [];
    $rows[] = [
        'mac'      => $mac,
        'hostname' => $node['computername'] ?: $c['hostname'] ?: '(unknown)',
        'user'     => $node['pid'] ?? $c['username'] ?? '',
        'role'     => $c['user_profile_name'],
        'rssi'     => $c['rssi'],
        'snr'      => $c['snr'],
        'ssid'     => $c['ssid'],
        'vlan'     => $c['vlan'],
        'health'   => $c['client_health'],
        'pfStatus' => $node['status'] ?? '(no PF record)',
        'autoreg'  => $node['autoreg'] ?? 'no',
    ];
}

SSID Broadcast table (Wireless tab)

Two-call join keyed on SSID id.

$status   = $xiqClient->getSsidStatus($apXiqId);
// $status = { "70849780830896": {wifi0_enabled:true, wifi1_enabled:true}, … }

$policy   = $xiqClient->getDeviceNetworkPolicy($apXiqId);
$ssidList = $xiqClient->getPolicySsids($policy['id']);
// $ssidList[i] = { id, name, broadcast_name, access_security: { security_type, … } }

$rows = [];
foreach ($ssidList as $ssid) {
    $st = $status[(string)$ssid['id']] ?? [];
    $rows[] = [
        'name'      => $ssid['broadcast_name'],
        'security'  => $ssid['access_security']['security_type'],
        'wifi0'     => $st['wifi0_enabled'] ?? false,
        'wifi1'     => $st['wifi1_enabled'] ?? false,
        'wifi2'     => $st['wifi2_enabled'] ?? false,
    ];
}

Live Telemetry strip (Overview tab)

Three concurrent calls (curl_multi_init) drive the four sparklines:

  • XIQ /d360/wireless/interfaces-graph?channel=WIFI0 → channel-util(0) + connected-clients(0) sparklines
  • XIQ /d360/wireless/interfaces-graph?channel=WIFI1 → channel-util(1) + connected-clients(1) sparklines
  • XIQ /d360/client/graph?filter=CONNECTED_CLIENTS → AP-total client-count sparkline
  • ZBX history.get → CPU, memory, temperature, uplink in/out (existing portdetail/ pattern)

Sort each XIQ array by timestamp ascending before passing to JS (G9).
