→ Why This Document Exists

The original M0_PacketFence_Historical_API.html was written speculatively against the PF source code on GitHub, before any live API call was made. M1 task Validate PacketFence switch_ip filter + radius_audit_log turned the speculation into ground truth by running every PFClient method against the live PF 15 instance during school hours. This document captures the diff: what was right, what was wrong, and what the corrected query bodies actually look like.

Use this doc, not the original M0 PF doc, as the canonical reference when adding new PF endpoints in M2–M5. Schema columns and table semantics in the original doc are still correct (and worth re-reading), but every claim about endpoint paths, operator names, and filter syntax should be cross-checked against the findings table below.

1 Findings Summary

Every claim in the original M0 PF doc that affected PFClient implementation, with verdict and correction.

Topic Verdict Original claim → corrected
nodes/search filter Wrong
Filter nodes/search by switch_ip = AP_IP to get an AP's wireless clients.
→ switch_ip is not a column on the node table; PF returns HTTP 422 "switch_ip is an invalid field". Use PFClient::getActiveClientsForAp($apIp) which orchestrates locationlog → MAC list → nodes/search by mac.
datetime operator Wrong
Operator greater_than_or_equals for time-window filtering.
→ PF returns 422 "op is not valid". Use greater_than; PFClient compensates with $from − 1 seconds to preserve inclusive boundary semantics.
radius endpoint Wrong
Endpoint /api/v1/radius_audit_log (singular).
→ PF returns 404. Correct path: /api/v1/radius_audit_logs (plural) — consistent with nodes and locationlogs.

GET with ?filter=field:value compact query params.
→ PF returns 400 "schema: invalid path 'filter'". Use POST /api/v1/radius_audit_logs/search with the same structured query body shape as nodes/search and locationlogs/search.

Filter called_station_id starts_with $apMac — relies on Called-Station-Id format BSSID:SSID.
→ Use nas_ip_address starts_with $apIp. Vendor-independent (no BSSID-format guessing), consistent with the AP-IP-as-identifier pattern used by every other PF call in PFClient.

Field reply for the RADIUS Reply-Message attribute.
→ Schema column is radius_reply. Primary reason field is reason — confirmed populated. Fallback chain: reason → radius_reply → auth_status.

Parse SSID by splitting called_station_id on : after the BSSID.
→ radius_audit_logs has a dedicated ssid column. Read it directly. Parser kept as defensive fallback only.

Fetch retention via GET /api/v1/config/maintenance/locationlog_cleanup.
→ Path 404s. Measure empirically: query oldest locationlog row for the AP, derive age from start_time. Reflects real availability rather than configured maximum, and handles new APs gracefully.
— Still correct, kept for reference —
locationlog scoping Right
switch_ip is a valid column on the locationlog table. Filtering locationlogs/search by switch_ip = $apIp returns this AP's session history exactly as expected.
open-session sentinel Right
Open sessions carry end_time = '0000-00-00 00:00:00', not NULL. Treat this string literally when filtering or projecting.
PF ↔ XIQ agreement Right
Active session count from PF locationlog matched XIQ client count within ±2 during validation. Tolerance margin holds for the dashboard's display purposes.
bucket algorithm Right
PHP-side bucketing of locationlog sessions into a client-count time series produces the expected sparkline shape. PFClient::bucketClientCounts() with distinct-MAC dedupe is correct.

2 PFClient Method Reference (verified shapes)

The four operational methods callers should reach for, with the exact verified query body. Each block reflects what passes against PF 15 today — copy-paste safe.

Orchestration getActiveClientsForAp(string $apIp): array Verified

The Clients tab primitive. Two PF calls under the hood: locationlog filtered by switch_ip + open-session sentinel to produce a MAC list, then nodes/search filtered by an OR-of-equals on mac to enrich each MAC with registration data.

Returns merged rows: {mac, node: {...}, session: {...}}. Use this — never call nodes/search with a switch_ip clause directly.

POST /api/v1/locationlogs/search Verified

Returns {series, sessions, truncated}. series is the bucketed sparkline data, sessions is the raw rows for callers that need their own grouping.

{
  "cursor": 0,
  "limit":  1000,
  "sort":   [ "start_time ASC" ],
  "fields": [ "mac", "ssid", "start_time", "end_time", "role" ],
  "query": {
    "op": "and",
    "values": [
      { "op": "equals",       "field": "switch_ip",  "value": "172.16.97.59" },
      { "op": "greater_than", "field": "start_time", "value": "2026-05-05 23:59:59" }
    ]
  }
}

Note the −1s on the lower bound — compensation for switching from greater_than_or_equals (rejected by PF) to greater_than.

POST /api/v1/nodes/search Verified

Filter by mac only — never by switch_ip (that field doesn't exist on the node table). For multiple MACs, OR a list of equals clauses; the static helper PFClient::nodeSearchBody($clauses, $limit) builds this correctly.

{
  "cursor": 0,
  "limit":  100,
  "sort":   [ "mac ASC" ],
  "fields": [ /* device_class, last_seen, status, etc. */ ],
  "query": {
    "op": "and",
    "values": [{
      "op": "or",
      "values": [
        { "op": "equals", "field": "mac", "value": "a4:c3:f0:12:34:56" },
        { "op": "equals", "field": "mac", "value": "b8:27:eb:aa:bb:cc" }
        // … one clause per MAC in list
      ]
    }]
  }
}
POST /api/v1/radius_audit_logs/search Verified

Auth-failure rows scoped to one AP, grouped by client MAC with attempt counter. Endpoint is plural (radius_audit_logs, not radius_audit_log). Filter is by AP IP, not MAC.

{
  "cursor": 0,
  "limit":  100,
  "sort":   [ "created_at DESC" ],
  "fields": [
    "mac", "called_station_id", "ssid",
    "user_name", "stripped_user_name",
    "auth_status", "reason", "radius_reply",
    "created_at"
  ],
  "query": {
    "op": "and",
    "values": [
      { "op": "starts_with", "field": "nas_ip_address", "value": "172.16.97.59" },
      { "op": "equals",      "field": "auth_status",    "value": "Reject" }
    ]
  }
}

starts_with rather than equals on nas_ip_address handles the case where PF appends a port or interface suffix in some configurations.

3 PF 15 Idiosyncrasies — Things to Know Before Adding Endpoints

Non-obvious behaviour patterns that bit us during validation. Watch for these when implementing the next round of PF endpoints (Events tab merge, M5).

I-1 Empty results sometimes return HTTP 404, not 200 + {items: []}
radius_audit_logs/search returns HTTP 404 with body {"errors":[{"message":"entries not found"}]} when a valid query matches zero rows. locationlogs and nodes use the conventional 200 + empty items[]. The asymmetry isn't documented anywhere — discovered only by running the script. For every new search endpoint, treat 404 + "entries not found" as empty success, not as an error. PFClient lifts a small predicate for this; promote it to a shared helper if a second endpoint shows the same pattern.
I-2 No greater_than_or_equals operator
PF 15's filter DSL has equals, not_equals, contains, starts_with, ends_with, greater_than, less_than — and that's it. No _or_equals compound operators. For inclusive boundaries, subtract or add 1 unit on the value side. PFClient compensates for datetime windows; do the same for any new range-filter implementation.
I-3 Endpoint paths are plural; /<entity>/search is the universal POST shape
Every search endpoint is POST /api/v1/<entity>s/search with the same query-body shape ({cursor, limit, sort, fields, query: {op, values: [...]}}). GET ?filter=… compact syntax is rejected with HTTP 400. When implementing a new endpoint, default to POST /search first; only consider GET if the operation is clearly RPC-style (login, config-fetch).
I-4 switch_ip exists on locationlog, not on node
The node table is the device registration store — no AP context. The link between "device" and "AP" lives only in locationlog, which is why every AP-scoped client query is a two-step: locationlog finds the MAC, nodes/search enriches it. When the Events tab is built in M5, expect the same pattern wherever AP-scoping intersects device data.
I-5 The schema discovery page is the fastest way to verify field names
Hitting any /api/v1/<entity>s/search endpoint with no body (or via the PF API explorer UI) returns a sample query object enumerating every valid field name — not by listing operators, but by populating each field with "foo". This is gold for validating field names before writing query code; it's how this validation cycle compressed several hours of trial-and-error into a few targeted edits.

4 Validation Harness — How to Re-Run

The script test_pf_validation.php exercises every PFClient public method against a live PF instance and reports pass/fail per check. Re-run it any time PF is upgraded, the webservices role changes, or PFClient is modified.

Invocation

# From modules/apdetail/includes/ on a host with network access to PF
PF_URL=https://PF_HOST:9999 \
PF_USER=webservices \
PF_PASS='…' \
AP_IP=172.16.97.59 \
XIQ_CLIENT_COUNT=23 \
php test_pf_validation.php

Exit codes

0 = all required checks passed (M1 unblocked) · 1 = at least one required check failed · 2 = config error (missing env vars).

Check structure

Five checks, two required for M1 gating:

#Required?What it verifies
1RequiredgetActiveClientsForAp returns enriched client rows during school hours.
2RequiredlocationlogSearch returns sessions and produces a usable bucket series.
3OptionalActive session count agrees with operator-supplied XIQ client count within ±2.
4OptionalradiusAuditLog filter format works (PASS-with-WARN if AP has no recent failures).
5OptionalEmpirical retention measurement — informs M2 time-range selector.
Run during school hours. Empty results outside class periods are ambiguous (filter broken vs. AP genuinely idle). The script emits a WARN when run outside 08:00–15:00 weekdays, but a definitive pass requires real load on the AP.

5 Outstanding for M2+

Things that are not blockers for M1 but warrant attention as the dashboard work continues.

Force a real auth failure during M4 testing to definitively confirm radiusAuditLog's filter is working. The current PASS-with-WARN state ("zero failures returned, AP may genuinely have none") is technically ambiguous. Manually fail an 802.1X auth on this AP, then re-run the validation; the script should pick up the deliberate failure within seconds and convert WARN to PASS-with-data.
Watch for retention truncation at 1000 rows. Check 5 measures retention by inspecting the oldest row in a 1000-row sample. If the script reports truncated: true, real retention extends beyond the sample window. For BHS-56-Hallway during the school year, ~50 unique clients × ~5 sessions/week ≈ 4 weeks before truncation kicks in. M2's time-range selector should handle the truncated case gracefully (offer a "more" option that triggers paginated retrieval).
Two PF data shapes still unverified because they didn't surface during M1 validation: nas_ip_address port-suffix scenarios (the starts_with operator handles these by design, but no live data has yet shown a suffix), and the format of called_station_id when the AP emits non-standard NAS-Identifier values. Both are defensively handled in PFClient; flag them if M4 client testing surfaces unexpected behaviour.
The "entries not found" 404 idiom may appear on other endpoints. Currently only confirmed on radius_audit_logs/search. If M5's Events tab work hits the same pattern on audit_log (the unified PF audit log), promote the detection predicate from PFClient::radiusAuditLog into a shared helper. The check is small enough that duplicating once is fine; the moment a second endpoint exhibits it, refactor.
