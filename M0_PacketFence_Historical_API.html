→ Why This Document Exists

The M0 XIQ API Reference concluded that ExtremeCloud IQ is point-in-time only — it cannot provide history curves for channel utilization, noise floor, or client count. PacketFence's locationlog table is a timestamped record of every RADIUS association event, stored permanently (subject to the configured cleanup window). Queried with the right filters, it yields a genuine historical time series for client population on a specific AP — the one sparkline the Live Telemetry strip couldn't otherwise get.

Zabbix
Full TSDB history for uplink traffic, latency, packet loss, CPU, memory, temp, PoE. Drives sparklines directly via history.get.
XIQ
Point-in-time snapshots only. Channel utilization, noise floor, TX power. No time-series history available — show as current-value badge.
PacketFence
Historical association records. Every connect/disconnect event timestamped. Derive: clients-connected timeline, auth failure history, SSID utilization trend.
Net result: The Live Telemetry sparkline strip gains one historically-backed metric from PF: Clients Connected (count over time). All other wireless RF metrics (channel util, noise) remain current-value-only badges. This is better than the fallback of adding new SNMP items to the Zabbix template.

1 locationlog Table — Full Schema

The locationlog table is PF's core association log. Every RADIUS Access-Accept or accounting event for a wireless client creates or updates a row. The schema is confirmed from the PF source and GitHub issue #5029 (INSERT statement, all columns enumerated).

Open/active session sentinel: PF writes end_time = '0000-00-00 00:00:00' (not NULL) for sessions that have not yet ended. All PHP queries filtering for currently-active sessions must use end_time = '0000-00-00 00:00:00', and all historical queries must account for this value meaning "still open."

2 Locationlog API Endpoints

POST /api/v1/locationlogs/search Historical — time-range aware
The primary endpoint for dashboard use. Accepts a structured JSON query with field filters, sort order, field selection, and a pagination limit (max 1000). Returns all association events for a given AP within a time window — the raw data for client-count sparklines and SSID distribution history.

GET /api/v1/radius_audit_log Auth failure history
Already in the project plan (M1 validation task). Included here for completeness. The radius_audit_log complements locationlog: where locationlog records successful associations (Access-Accept), the RADIUS audit log captures all RADIUS events including Access-Reject (auth failures). Filter by called_station_id = AP BSSID (radio MAC, format: 74:26:AC:2C:A4:90:TCS-Students — note SSID is appended in Called-Station-Id for wireless). The Auth Failures table on the Clients tab is driven by this endpoint.
called_station_id format for wireless: In RADIUS for 802.11 APs, Called-Station-Id is formatted as BSSID:SSID (e.g. 74:26:AC:2C:A4:90:TCS-Students). The filter must use starts_with on the AP MAC prefix, not an exact equals on the switch_mac, to catch all SSIDs. Validate the exact format on BHS-56-Hallway during M1. 
// Auth failure query for Auth Failures tab
GET /api/v1/radius_audit_log

  &filter=auth_status:Reject
  &sort=created_at DESC
  &limit=100

// Fields to request: mac, called_station_id, user_name, auth_status,

3 Deriving the Client Count Sparkline

The key insight: each locationlog row has a start_time and an end_time. At any given moment T, a session was active if start_time ≤ T AND (end_time = '0000-00-00 00:00:00' OR end_time ≥ T). By iterating over time buckets within the dashboard time window, you get a client count series — a proper sparkline with real historical shape.

Step 1
Fetch locationlogs
POST /locationlogs/search
Filter: switch_ip = AP_IP and start_time ≥ window_start.
Max 1000 records.
Step 2
Bucket in PHP
Divide time window into N buckets (e.g. 12 × 5-min for 1h, 24 × 30-min for 12h). For each bucket midpoint T, count sessions where start ≤ T ≤ end.
Step 3
Emit sparkline data
Return [{ts, count}] array. JS renderer (adapted from portdetail sparkline) draws the curve. Label: "Clients connected (PF)".
Step 4
Time-period sync
Re-fetch when _timeperiod broadcast changes. Same pattern as existing portdetail Zabbix sparklines — same event handler.

PHP bucketing implementation sketch

/**
 * Build a client-count time series from locationlog records.
 *
 * @param array  $sessions  Array of locationlog rows: [mac, start_time, end_time]
 * @param int    $from      Unix timestamp — window start
 * @param int    $to        Unix timestamp — window end
 * @param int    $buckets   Number of data points to emit (e.g. 24)
 * @return array            [{ts: int, count: int}]
 */
function buildClientSpark(array $sessions, int $from, int $to, int $buckets = 24): array {
    $step   = (int)(($to - $from) / $buckets);
    $series = [];

    for ($i = 0; $i < $buckets; $i++) {
        $t     = $from + ($i * $step) + ($step / 2);   // bucket midpoint
        $count = 0;

        foreach ($sessions as $s) {
            $start = strtotime($s['start_time']);
            $end   = $s['end_time'] === '0000-00-00 00:00:00'
                ? PHP_INT_MAX            // still open → active until now
                : strtotime($s['end_time']);

            if ($start <= $t && $end >= $t) {
                $count++;
            }
        }

        $series[] = ['ts' => (int)$t, 'count' => $count];
    }

    return $series;
}
Performance note: This is an O(N × B) loop in PHP where N = locationlog records and B = buckets. At N=500 and B=24, that's 12,000 comparisons — negligible. If the AP is pathologically busy and returns 1000 records, the loop is still only 24,000 iterations, well within a single widget render budget.

5 Caching Strategy + Retention Warning

Cache TTL for locationlog queries

Unlike XIQ config data (SSIDs, policy names — cache 5 min), locationlog queries represent session state that changes frequently. Do not cache aggressively:

Query typeCache TTLReason
Client count sparkline (historical window)60 sHistorical data doesn't change — safe to cache. Matches widget refresh interval.
Currently active sessions (end_time = 0)30 sSame PF node search already cached at 30 s in existing widget — match that.
SSID client counts (active sessions grouped)30 sClients connect/disconnect frequently during class transitions.
Per-MAC session history (/node/{mac}/locationlogs)No cacheOnly called on operator click — live response expected.

Retention window — validate before M2

The locationlog has a configurable cleanup window (maintenance.locationlog_cleanup_window in PF config). The default is often 30 days, but it can be shorter. If TCS has set a shorter window, historical data beyond that point is gone. Check the current retention setting on the TCS PF 15 instance before building sparklines that depend on multi-day history: 
# On PF server
grep locationlog_cleanup /usr/local/pf/conf/pf.conf
# or via API:
GET /api/v1/config/maintenance/locationlog_cleanup
If the retention window is less than 24 hours, the sparkline can only cover the school day's own data. That may still be sufficient for the dashboard's use case (show today's pattern) but cap the _timeperiod selector at the configured retention window.

Auth endpoint + session management

Reuse the existing PF widget auth pattern exactly. The existing packetfence/ widget already manages PF session auth (POST /api/v1/login → token → session cookie or Bearer header). All locationlog calls go through the same PFClient.php (or equivalent) HTTP client. No new auth logic — just new endpoint calls on the existing authenticated session.
